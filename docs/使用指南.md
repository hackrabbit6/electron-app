# 使用指南

一份极简、可定制的 Electron 桌面应用模板。核心(主进程 / preload)与前端框架解耦,通过一条命令即可在 **React / Vue / Svelte / Vanilla TS** 之间切换。

- ⚡ Electron 37 + Vite 7 + TypeScript
- 🧩 一条命令切换前端框架(`pnpm scaffold`)
- 🪶 不绑定任何 CSS 框架,只有一个验证 preload 桥接的极简欢迎页
- 🔒 默认安全配置:开启 `contextIsolation`、关闭 `nodeIntegration`、设置 CSP、外部链接走系统浏览器

---

## 一、环境要求

| 工具 | 版本 |
| --- | --- |
| Node.js | ≥ 18(建议 20+) |
| pnpm | ≥ 9(本模板用 11 验证过) |
| 操作系统 | 开发跨平台;打包脚本目前面向 macOS |

> 没装 pnpm 的话:`npm install -g pnpm`。

---

## 二、快速开始

```bash
pnpm install     # 安装 Electron 核心依赖
pnpm scaffold       # 交互式选择前端框架(写入渲染层 + 依赖)
pnpm dev         # 启动应用,带热重载
```

仓库**默认已预装 React**,所以克隆下来直接 `pnpm dev` 就能跑。想换框架时再运行 `pnpm scaffold`。

> ⚠️ 用 `pnpm scaffold`,**不要用 `pnpm setup`** —— `setup` 是 pnpm 的保留内置命令,会被 pnpm 拦截并报 `Unknown option`。如果你习惯叫它 setup,用 `pnpm run setup`(带 `run`)也可以。

启动后会弹出一个 900×700 的窗口,显示「⚡ Electron + React」欢迎页,并列出当前的 Electron / Node / Chrome 版本 —— 这些版本号是通过 preload 桥接拿到的,能跑出来就说明主进程 ↔ 渲染进程通了。

---

## 三、选择 / 切换前端框架

`pnpm scaffold` 是一个**零依赖的交互式脚手架**(只用 Node 内置模块)。它会把 `templates/` 里选中的框架变体复制到 `apps/desktop/`,并重新生成 `package.json`。

### 交互式(推荐)

```bash
pnpm scaffold
```

会出现菜单:

```
Choose a frontend framework:

  1) react
  2) svelte
  3) vanilla
  4) vue

Enter number [1-4]:
```

输入编号或框架名即可。如果当前已有渲染层,会先提示确认是否覆盖。

### 非交互式

```bash
pnpm scaffold vue                 # 直接指定框架
pnpm scaffold svelte --no-install # 跳过自动 pnpm install
pnpm scaffold react --yes         # 跳过覆盖确认
```

支持的框架:`react`、`vue`、`svelte`、`vanilla`。

> ⚠️ **切换框架会覆盖当前渲染层**(`src/`、`index.html`、`vite.config.ts`、`tsconfig.json`)。切换前请先提交你的代码。交互模式会先弹确认。

切换时脚手架做了四件事:
1. 清掉旧渲染层文件;
2. 复制新框架的模板文件到 `apps/desktop/`;
3. 用 `templates/_base.package.json` + 该框架的 `deps.json` 重新生成 `package.json`(含正确的类型检查工具);
4. 在 `package.json` 里记录 `electronTemplate.framework`,并自动 `pnpm install`(除非加了 `--no-install`)。

## 四、日常命令

> 都在仓库根目录运行,会通过 pnpm 转发到 `desktop` 包。

| 命令 | 作用 |
| --- | --- |
| `pnpm scaffold` | 选择 / 切换前端框架 |
| `pnpm dev` | Vite 开发服务器 + Electron,热重载,自动开 DevTools |
| `pnpm build` | 生产构建 → `dist/`(渲染层)+ `dist-electron/`(主进程、preload) |
| `pnpm typecheck` | 类型检查(按框架自动用 `tsc` / `vue-tsc` / `svelte-check`) |
| `pnpm dist:mac` | 先构建,再用 electron-builder 打包成 macOS `.dmg` / `.zip` |

本地打**未签名**的 mac 包:

```bash
pnpm --filter desktop dist:mac -- --unsigned
```

> 模板未配置测试框架和 lint 工具,可按需自行添加。

## 五、项目结构

```
electron-app/
├── templates/                 # 框架模板库(脚手架的数据源,放在仓库根目录)
│   ├── _base.package.json     # 基础 package.json;setup 会合并各框架的 deps.json
│   ├── react/  vue/  svelte/  vanilla/
│   └── …/{ index.html, vite.config.ts, tsconfig.json, src/, deps.json }
├── apps/desktop/
│   ├── main/index.ts          # Electron 主进程(框架无关)
│   ├── preload/index.ts       # contextBridge → window.electronAPI(框架无关)
│   ├── types/electron-api.d.ts# preload 桥接的共享类型
│   ├── scripts/
│   │   ├── setup.mjs           # 交互式脚手架
│   │   └── dist-mac.mjs        # macOS 打包脚本
│   ├── package.json            # 由 setup 生成
│   └── (运行 setup 后生成的渲染层:index.html / vite.config.ts / tsconfig.json / src/)
├── package.json               # 工作区脚本
├── pnpm-workspace.yaml         # 工作区 + 构建脚本批准(allowBuilds)
└── README.md
```

> **为什么 `templates/` 放在仓库根目录而不是 `apps/desktop/` 里?**
> 如果放在项目内,`tsc` / `svelte-check` / Vite 在加载配置时会扫描到其他框架的文件而报错(尤其 svelte-check 会尝试加载它找到的每个 `vite.config.ts`)。放在项目扫描范围之外是必须的,请勿移回去。

## 六、自定义

### 1. 改渲染层(你的 App 界面)

直接编辑 `apps/desktop/` 下的活动文件,例如 React 改 `apps/desktop/src/App.tsx`。

> 想改某个框架**未来脚手架生成**的默认内容,则改 `templates/<框架>/` 下的对应文件。

### 2. 改窗口 / 应用行为

编辑 `apps/desktop/main/index.ts`,可以调整窗口尺寸、标题、菜单、链接处理等:

```ts
mainWindow = new BrowserWindow({
  width: 900,        // 窗口宽
  height: 700,       // 窗口高
  minWidth: 600,
  minHeight: 500,
  title: 'Electron App',  // 改成你的应用名
  webPreferences: {
    preload: join(__dirname, '../preload/index.js'),
    contextIsolation: true,   // 安全:保持开启
    nodeIntegration: false    // 安全:保持关闭
  }
})
```

### 3. 扩展 preload API(主进程 ↔ 渲染进程通信)

当前 [preload/index.ts](../apps/desktop/preload/index.ts) 只暴露了平台和版本信息。要新增能力,推荐用 `ipcRenderer.invoke` + `ipcMain.handle` 模式。

**① preload 暴露方法** —— `apps/desktop/preload/index.ts`:

```ts
import { contextBridge, ipcRenderer } from 'electron'

contextBridge.exposeInMainWorld('electronAPI', {
  platform: process.platform,
  versions: { /* …保持原样… */ },
  // 新增:读取文件
  readFile: (path: string) => ipcRenderer.invoke('read-file', path)
})
```

**② 主进程实现处理器** —— `apps/desktop/main/index.ts`:

```ts
import { ipcMain } from 'electron'
import { readFile } from 'fs/promises'

ipcMain.handle('read-file', (_event, path: string) => readFile(path, 'utf-8'))
```

**③ 补充类型** —— `apps/desktop/types/electron-api.d.ts`(让所有框架变体都有类型提示):

```ts
declare global {
  interface Window {
    electronAPI: {
      platform: string
      versions: { node: string; chrome: string; electron: string }
      readFile: (path: string) => Promise<string>   // 新增
    }
  }
}
```

**④ 渲染层调用**:

```ts
const text = await window.electronAPI.readFile('/path/to/file.txt')
```

### 4. 改应用元信息 / 打包配置

应用名(`productName`)、应用 ID(`appId`)、electron-builder 的 `build` 配置都在 [templates/_base.package.json](../templates/_base.package.json) 里 —— **改这里**,这样重新跑 setup 也不会丢失:

```json
{
  "build": {
    "appId": "com.example.electronapp",
    "productName": "ElectronApp",
    "mac": { "identity": null, "hardenedRuntime": false },
    "target": ["dmg", "zip"]
  }
}
```

> 注意:`apps/desktop/package.json` 是 setup 生成的产物,直接改它会在下次切换框架时被覆盖。

### 5. 新增一个框架

在 `templates/` 下新建一个文件夹(比如 `solid/`),放入 `index.html`、`vite.config.ts`、`tsconfig.json`、`src/`、以及一个 `deps.json`。`pnpm scaffold` 会自动识别它。

`deps.json` 格式:

```json
{
  "dependencies": { "solid-js": "^1.9.0" },
  "devDependencies": { "vite-plugin-solid": "^2.11.0" },
  "scripts": { "typecheck": "tsc --noEmit" }
}
```

> `scripts` 是可选的覆盖项(比如 Vue 用 `vue-tsc`、Svelte 用 `svelte-check`)。每个 `vite.config.ts` 都**内联**了 Electron 主进程 / preload 插件(而不是引用共享模块),复制现有任一框架的配置改改即可。

## 七、打包发布(macOS)

```bash
pnpm dist:mac                              # 签名构建(若配置了证书)
pnpm --filter desktop dist:mac -- --unsigned   # 未签名,本地测试用
```

产物输出到 `apps/desktop/release/`,包含 `.dmg` 和 `.zip`。

> ⚠️ **首次打包需要联网。** electron-builder 会按需下载 Electron 包和 dmg 打包工具(`dmgbuild`)到本地缓存(`~/Library/Caches/electron/`)。网络受限时会卡在这一步(报 `EOF` / `ECONNRESET`):`.zip` 不依赖额外工具、通常先成功产出,`.dmg` 还需要 `dmgbuild` 下载成功。下载过的工具会进缓存,之后离线也能打包。

打包脚本 [scripts/dist-mac.mjs](../apps/desktop/scripts/dist-mac.mjs) 会:
- 读取当前 git 短 SHA 作为构建版本号(不在 git 仓库时回退为 `dev`);
- `--unsigned` 时关闭代码签名和 hardened runtime,方便本地出包。

> 想支持 Windows / Linux:在 `templates/_base.package.json` 的 `build` 里加 `win` / `linux` 配置,并参考 dist-mac.mjs 写一个对应脚本。

## 八、安全特性

模板默认采用 Electron 官方推荐的安全配置:

- **`contextIsolation: true`** —— 渲染进程与 preload 上下文隔离;
- **`nodeIntegration: false`** —— 渲染进程拿不到 Node API,只能通过 `window.electronAPI` 访问被显式暴露的能力;
- **CSP** —— 每个 `index.html` 都设置了内容安全策略;
- **外部链接** —— `http(s)` 链接和 `window.open` 一律交给系统浏览器打开,不在应用内打开。

扩展功能时请保持这些设置,新能力一律通过 preload 显式暴露。

## 九、常见问题

**Q:`pnpm dev` 报找不到 vite.config.ts / 渲染层文件?**
还没运行过 `pnpm scaffold`。先 `pnpm scaffold` 选一个框架再 `pnpm dev`。

**Q:安装时提示 `Ignored build scripts: electron, esbuild`,`pnpm dev` 直接退出?**
pnpm 默认拦截依赖的安装脚本(postinstall),导致 Electron 二进制没下载;而 `pnpm dev` 启动前会做依赖检查,一旦发现被拦截就报错退出。本模板已在 [pnpm-workspace.yaml](../pnpm-workspace.yaml) 里批准了这些脚本(`allowBuilds` 给本环境的 pnpm 11.x,`onlyBuiltDependencies` 给上游 pnpm,两个都留着)。如果克隆后仍提示被拦截,运行一次:

```bash
pnpm approve-builds --all
```

**Q:报 `Electron failed to install correctly`,但 `pnpm install` 看起来正常?**
这是"半安装"死锁:`dist/Electron.app` 已经解压,但标记文件 `path.txt` 缺失,于是 Electron 找不到二进制。删掉残留重新触发它的安装脚本即可:

```bash
ED=node_modules/.pnpm/electron@*/node_modules/electron
rm -rf $ED/dist $ED/path.txt && node $ED/install.js
```

**Q:切换框架后我之前的代码没了?**
`pnpm scaffold` 会覆盖渲染层。切换前务必提交 / 备份。交互模式会先确认。

**Q:类型检查报错说找不到 `window.electronAPI` 上的某个方法?**
在 [apps/desktop/types/electron-api.d.ts](../apps/desktop/types/electron-api.d.ts) 里补上对应类型(见「扩展 preload API」第 ③ 步)。

**Q:想改窗口默认大小 / 应用名?**
窗口看 `apps/desktop/main/index.ts`;应用名 / 打包信息看 `templates/_base.package.json`。
